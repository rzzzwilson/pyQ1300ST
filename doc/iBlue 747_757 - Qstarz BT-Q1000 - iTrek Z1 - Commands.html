RuleDescription of Log Format in logger memory

log::=log_block*//
//		A log is a series of blocks each of 0x10000 bytes). The first version of the iBlue has 32 blocks (2MB) that can be filled with 1.75 MB of usefull data (0.25MB goes in the headers of the blocks). On more recent versions and similar devices, the size of the memory got bigger but the block size is still the same.

.

log_block::=log_header log_chunk//A block in the log contains a header (0x200 bytes) and a packet of data (0xFE00 bytes)

.

log_header::=header_info
sector_status
454 bytes		//
//
//
//
//		header_info : explained below

Next: 454 bytes of FF
Ending with:
0x2AFFBBBBBBBB

.

header_info::=20 bytes//
//
//
//
//
//
//
//
//
//
//
//		Log info consist of 20 bytes, the log count is FFFF until the entire 64kByte block has been filled. It will then represent the number of log items in this block.
FFFF BBBBBBBB MMMM PPPP 0000 DDDD 0000 SSSS 0000
FFFF --- log count, updated when entire 64kB block full.
BBBBBBBB -- log format - LSB is first. In my case (reordered): 0x0020003F
MMMM -- Mode - 0401 --- ??? / also 0601 (second block in my case: what setting could that be? May be to indicate if the log period/distance/speed is active! (default=1s, my case: distance too)
PPPP -- Log period in 10:ths of seconds
DDDD -- Log distance in 10:ths of meters
SSSS -- Log speed in 10:ths of km/h

.

sector_status::=32 bytes//Indicates bad flash sectors - not sure how to interpret

.

log_chunk::=( log_packet
| sep_record )
chunk_fill		//Size is 0xFE00 bytes

.

chunk_fill::=0xFF *//End of chunk is filled with FF (flash is not written)

.

log_packet::=log_data
'*' log_checksum		//Each packet contains the data followed by '*' (0x2A) and the checksum of log_data

.

log_checksum::=<2 hex chars>//Two ASCII characters representing the hex value of the checksum

.

log_data::=utc_data ?!
valid_data ?!
latitude_data ?!
longitude_data ?!
height_data ?!
speed_data ?!
heading_data ?!
dsta_data ?!
dage_data ?!
pdop_data ?!
hdop_data ?!
vdop_data ?!
nsat_data ?!
( nosat_data
| sat_data * ) ?!
rcr_data ?!
mili-second_data ?
distance_data ?!		//
//
//
//
//
//
//		This defines the order of the data logged. The '?' indicates that the data is optional.
The actual data logged depends on the settings of the device at the moment of the log
[A '!' after the '?' indicates that the position of this data in the log has been confirmed.
Some information concerning size catched on http://pc11.2ch.net/test/read.cgi/mobile/1173306239

If there is no satellite data and SID is being logged, then the number of sats will be set to 0 and there will be no ele, azi or snr data)

.

(seconds since 1 jan 1970).

sep_record::=0xAAAAAAAA 0xAAAAAA 0xYY 0xXXXXXXXX 0xBBBBBBBB//
//
//
//
//
//
//
//
//
//
//
//		Switching the device on/off results in some synchronization ???
The information record is 16 bytes long and contains a command (YY) and an argument XXXXXXXX. The argument should be interpreted as a word or long depending on the command byte. LSB is first byte of the data XXXXXXXX. The following commands are known.
0x02 - Log bitmask change [long bitmask]
0x03 - Log period change [word period/10 sec]
0x04 - Log distance change [word distance/10 m]
0x05 - Log speed change [word speed/10 km/h]
0x06 - Log overwrite/log stop change
- argument = same as log status (P¨MTK182,2,7 response)
0x07 - Log on/off change
- argument = same as log status (P¨MTK182,2,7 response)
